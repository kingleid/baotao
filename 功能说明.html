2 两大部署模式
模式 A

裸 SSH 部署 (pm2 + Nginx)

通过 SSH 上传源码，用 pm2 守护进程运行应用，再用 Nginx 做反向代理将 80 端口映射到应用端口。适合快速联调、临时部署。

特点：不依赖宝塔面板 API，完全通过 SSH 操作，速度快、可控性强。

模式 B

面板原生托管项目

通过宝塔内部 Python 模型将应用注册为面板托管的 Node 项目，在面板 UI 中可见、可启停/重启、支持域名管理。

特点：面板可视化管理，与其他项目隔离，支持官方域名绑定。

核心决策：用 SSH，不用面板 API

宝塔面板 API 有 IP 白名单（L7 层），未白名单的 IP 访问任何路径都返回 nginx 404（不是真的 404）。SSH（端口 22）通常不在白名单范围内，是可靠路径。面板原生项目也通过 SSH 调用宝塔内部 Python（localhost 调用，绕过 HTTP 白名单）。

3 模式 A：裸部署工作流
0

收集连接信息

SSH 主机/端口/用户名、认证方式（密钥优先）、服务器 Web 根目录、应用监听端口（默认 3000）、健康检查端点（默认 /healthz）

1

环境探测 ssh_probe.py

只读探测：操作系统、磁盘、宝塔 Node 路径、Nginx 二进制、vhost 目录、监听端口、已有 pm2 进程。可同时追加本地部署公钥。

2

全量部署 deploy.py

上传源码（排除 node_modules/data/.git/.env）→ npm install → 生成 ecosystem.config.js（随机 SESSION_SECRET）→ pm2 启动+开机自启 → 健康检查

3

Nginx 反代 deploy_nginx.py

将 80 端口代理到 127.0.0.1:应用端口。支持 IP 直连模式（无域名）和域名+SSL 模式。写入面板真实 vhost 目录，nginx -t 通过后才 reload。

4

增量更新 deploy_update.py

代码改动后快速更新：重新上传源码 + pm2 restart，跳过耗时的 npm install。

5

验证

curl 检查 HTTP 200、测试登录与会话保持（cookie jar 验证代理后 session 不丢）

4 模式 B：面板原生项目注册

register_node_project.py — 一键完成

通过 SSH 调用宝塔自带 Python 解释器 /www/server/panel/pyenv/bin/python，直接运行面板内部模型类 projectModel.nodejsModel.main()，在 localhost 层面完成项目注册，完全面板原生。

1

上传源码

排除 node_modules/data/.git/.env，上传到 /www/wwwroot/<域名>/

2

npm install

用宝塔自带的 Node（/www/server/nodejs/<ver>/bin/npm）安装生产依赖

3

create_project

用安全名（^\w+$ 格式，不带点）创建项目，bind_extranet=0 先不绑外网

4

改名为域名

绕过 ^\w+$ 限制：改 sites 表 name + project_config + 重命名 pid/sh/log 三个文件

5

绑域名

project_add_domain 登记域名 → bind_extranet 开启外网映射 → 自动生成 node_<name>.conf 反代配置

使用示例
SSH_HOST=110.42.209.211 SSH_USER=root SSH_PWD='<密码>' \
LOCAL_DIR=/local/markdown-editor REMOTE_DIR=/www/wwwroot/markdown.example.com \
DOMAIN=markdown.example.com APP_NAME=md_editor REMOTE_PORT=3100 \
python scripts/register_node_project.py

7 实战踩坑清单（9 项）
🔒 宝塔面板 API 返回 404 — IP 白名单拦截
未白名单的 IP 访问面板任何路径都返回 nginx 404（不是超时）。看起来像路径错误，实际是 IP 被拦截。
✅ 不走面板 API，改用 SSH（端口 22 通常不在白名单内）。面板原生项目也通过 SSH 调用内部 Python。
📁 Nginx vhost 真实目录
手写在 /www/server/nginx/conf/vhost/ 的配置不生效。宝塔 nginx.conf include 的是 /www/server/panel/vhost/nginx/。
✅ vhost 文件写入 /www/server/panel/vhost/nginx/<name>.conf，然后 nginx -t && reload。
🚫 云安全组封锁应用端口 (3000)
本地 curl 127.0.0.1:3000 正常，公网访问超时。云安全组只放行 80/443/22/8888。
✅ Nginx 代理 80 → 127.0.0.1:3000，利用已开放的 80 端口，无需改安全组。
👻 pm2 孤儿进程
pm2 startup 后 pm2 jlist 为空，但旧 node 进程仍在服务。重启后可能无法恢复。
✅ pm2 kill → pm2 start ecosystem.config.js → pm2 save → 验证 jlist 中有应用名 → pm2 startup。
⚔️ 不要同时用面板和 pm2 启动同一应用
两个管理器抢同一端口，面板启动失败或冲突。
✅ 只用一个管理器。选 pm2 则不在面板点"启动"；选面板则先停 pm2 + 删 vhost conf。
🔄 前端 401 → reload 死循环
api() 收到 401 就 location.reload()，导致刷新→未认证→401→刷新→无限循环。
✅ 401 时跳转登录页而非 reload。设置 cookie: httpOnly; sameSite=lax。用户需硬刷新清除缓存。
📛 项目名 ^\w+$ 限制
create_project 拒绝带点的名字（如域名 markdown.example.com）。
✅ 先用安全名（md_editor）创建，再改 sites 表 name + project_config + 重命名 pid/sh/log 文件。
🔌 面板不注入 PORT 环境变量
宝塔 start_project 不会设置 PORT，应用可能落到默认 3000 与其他项目冲突。
✅ server.js 必须自己兜底：const PORT = process.env.PORT || 3100;
🗃️ sqlite3 CLI 读不到新行 (WAL)
直接用 sqlite3 命令行读 default.db 可能读不到刚写入的行（WAL 未落盘）。
✅ 用面板自己的 public.M() 读取：public.M('sites').where(...).select()
8 安全须知
⚠️ 密码与密钥安全

SSH 密码仅通过环境变量在部署时传递，绝不写入项目文件或 memory。优先使用密钥认证：本地生成 ed25519 密钥对，将公钥交给用户追加到 authorized_keys，无需交换秘密。

⚠️ SESSION_SECRET 生成

部署时由服务器端 os.urandom(24).hex() 生成随机密钥，不硬编码。写入 ecosystem.config.js 的 env 中。
